Hoy validamos ANTES de diseñar interfaces completas

🌳 ¿Qué es Tree Testing?

Tree Testing es una técnica para validar si tu sitemap es intuitivo ANTES de diseñar interfaces visuales.

¿Cómo funciona?

Le das al usuario una tarea ("Encuentra dónde cambiar tu contraseña") y le muestras el sitemap como un árbol de texto (sin diseño visual).

El usuario navega el árbol hasta donde cree que está esa función.

Resultado:

  • Si llega al lugar correcto → ✅ Sitemap funciona
  • Si NO llega o se confunde → ❌ Sitemap necesita ajustes

🎯 ¿Para Qué Sirve Tree Testing?

✅ Propósito principal:

Validar que tu arquitectura de información es intuitiva ANTES de invertir tiempo en diseñar pantallas.

Pregunta que responde:

"Si un usuario busca [X], ¿puede encontrarlo fácilmente en mi estructura?"

Ventajas:

  • Detectar problemas de navegación TEMPRANO (cuando es barato arreglarlos)
  • Validar nombres de categorías
  • Identificar dónde usuarios se pierden
  • Tomar decisiones basadas en datos, no opiniones

🔀 Tree Testing vs Card Sorting

Aspecto Card Sorting (A4) Tree Testing (HOY)
Propósito CONSTRUIR estructura VALIDAR estructura
Input Cards sueltas (features) Sitemap completo
Output Categorías + Sitemap Métricas de éxito
Usuario hace Agrupar y nombrar Buscar y encontrar
Cuándo ANTES de tener sitemap DESPUÉS de tener sitemap

Secuencia correcta: Card Sorting → Sitemap → Tree Testing → Wireframes

📅 ¿Cuándo Hacer Tree Testing?

1Situación ideal:

Después de Card Sorting, cuando tienes sitemap draft pero ANTES de diseñar pantallas

2También útil para:

  • Rediseño de navegación existente
  • Comparar 2 estructuras diferentes (A vs B)
  • Validar cambios de nombres de categorías

NO hacer Tree Testing si:

  • No tienes sitemap (primero Card Sorting)
  • Ya diseñaste todas las pantallas (muy tarde)
  • El sitemap tiene solo 2-3 opciones (demasiado obvio)

Tarea dada al usuario:

❌ Fracaso

📊 Métricas Clave de Tree Testing

Métrica Descripción Objetivo
Éxito directo Usuario llegó al lugar correcto en primer intento >70%
Éxito indirecto Llegó después de explorar/retroceder <20%
Fracaso NO llegó o llegó al lugar equivocado <10%
Tiempo promedio Segundos que tardó en completar tarea <30 seg
Primer clic ¿El primer clic fue correcto? >80%

Regla general: Si tasa de éxito total (directo + indirecto) es >70% → Sitemap funciona ✓

Si <60% → Necesita rediseño ❌

Árbol para Tree Testing:

Igual estructura, pero:

  • Solo texto (sin íconos, colores)
  • Nombres exactos del sitemap
  • Mismo orden
  • Misma jerarquía

Importante: NO cambies nombres en el árbol. Si el sitemap dice "Vista", el árbol dice "Vista".

✍️ Paso 2: Crear Tareas/Escenarios

Crear 5-8 tareas que representen acciones reales que los usuarios harían

¿De dónde salen las tareas?

Del Task Analysis (A3). Las tareas críticas que identificaron.

Estructura de una buena tarea:

Formato: "[Escenario]. ¿Dónde irías?"

Ejemplo: "Acabas de crear una tarea y quieres marcarla como completada. ¿Dónde irías?"

Características de buenas tareas:

  • Realistas: Situaciones que realmente pasan
  • Específicas: No ambiguas
  • Sin palabras del menú: No decir "quieres ir a Vista" (demasiado obvio)
  • Variadas: Cubren diferentes partes del sitemap

📝 Ejemplo: 6 Tareas para TaskSync

Tarea 1: "Necesitas conectar tu cuenta de Google Calendar con la app. ¿Dónde irías?"

Respuesta correcta: Sincronización → Importar Calendar

Tarea 2: "Quieres ver qué tienes programado para toda la semana. ¿Dónde irías?"

Respuesta correcta: Vista → Ver semanal

Tarea 3: "Te diste cuenta que hay un conflicto entre un turno de trabajo y una entrega. Quieres que la app te avise cuando esto pase. ¿Dónde irías?"

Respuesta correcta: Configuración → Notificaciones (o Alertas → Detectar conflictos)

Tarea 4: "Acabas de crear una tarea manual y ahora quieres marcarla como completada. ¿Dónde irías?"

Respuesta correcta: Tareas → Marcar completada

🔄 Pasos 3-5: Ejecutar y Analizar

3Reclutar participantes (5-10)

Mismo perfil que Card Sorting. Idealmente del A2.

4Conducir prueba

  • Dar 1 tarea a la vez
  • Usuario navega árbol hasta donde cree que está
  • NO ayudar ni dar pistas
  • Documentar: ¿Llegó? ¿Cuántos clics? ¿Cuánto tiempo?
  • Repetir con todas las tareas

5Analizar resultados

  • Calcular % de éxito por tarea
  • Identificar dónde fallan más
  • Si <60% éxito → rediseñar sitemap

✅ Checklist - Actividad Tree Testing

□ Árbol de texto creado (basado en sitemap A4)

□ 3 tareas creadas (realistas, del A3)

□ Tareas NO usan palabras exactas del menú

□ Participante navegó SIN ayuda

□ Documentaron: ✓/✗, ruta, tiempo por cada tarea

□ Si falló, identificaron POR QUÉ (nombre confuso, categoría mal ubicada, etc.)

🔀 De Tree Testing a Wireframes

Secuencia correcta:

Card Sorting → Sitemap → Tree Testing → Wireframes

¿Por qué Tree Testing primero?

  • Si el sitemap NO funciona → Arreglarlo ANTES de diseñar
  • Wireframes sin sitemap validado = trabajo desperdiciado
  • Tree Testing toma 2-3 horas. Wireframes toman días/semanas.

Ahora que validaron (o identificaron problemas):

Es momento de empezar a prototipar con Wireframes.

📱 ¿Qué son Wireframes?

Wireframes son representaciones de baja/media fidelidad de la estructura y contenido de tus pantallas.

Analogía:

Si el sitemap es el plano arquitectónico de una casa (dónde van las habitaciones), los wireframes son los planos detallados de cada habitación (dónde va cada mueble).

Propósito:

  • Definir estructura y layout de cada pantalla
  • Mostrar jerarquía de información
  • Planificar navegación entre pantallas
  • Validar flujos de usuario

NO son diseños finales. Son bocetos estructurales.

🎯 ¿Para Qué Sirven los Wireframes?

✅ SÍ son para:

  • Definir dónde va cada elemento
  • Mostrar jerarquía visual
  • Validar flujos con usuarios
  • Comunicar ideas a equipo
  • Iterar rápidamente

❌ NO son para:

  • Decidir colores finales
  • Elegir tipografía específica
  • Diseñar íconos detallados
  • Escribir copy final
  • Mostrar a clientes como "final"

Mantra: Wireframes son sobre estructura, no sobre estética.

📐 Niveles de Fidelidad

Nivel Descripción Cuándo usar
Low-fi Sketches en papel/pizarra. Muy rápidos. Grises, cajas, líneas. Brainstorming inicial, exploración rápida
Mid-fi Digital (Figma). Escala de grises. Estructura clara. Algunos detalles. Este curso - Validar con usuarios
Hi-fi Digital con colores, tipografía, imágenes reales. Casi diseño final. Después de validar Mid-fi (no en DEIU)

Para este curso: Trabajarán en Mid-fi (digital, gris, estructura clara)

🤔 ¿Cuál Nivel Usar?

Low-fi (Papel)

Usar cuando:

  • Explorando muchas ideas rápido
  • Brainstorming en equipo
  • No sabes qué va en la pantalla
  • Primeras 2-3 iteraciones

Tiempo: 5-10 min por pantalla

Mid-fi (Figma gris)

Usar cuando:

  • Ya sabes qué va en la pantalla
  • Quieres validar con usuarios
  • Comunicar a desarrolladores
  • Crear prototipos clicables

Tiempo: 30-60 min por pantalla

Hi-fi (Diseño final)

Usar cuando:

  • Mid-fi ya validado
  • Listo para handoff
  • Presentar a stakeholders
  • Última iteración

Tiempo: 2-4 hrs por pantalla

Error común: Saltar directo a Hi-fi. Pierdes flexibilidad y tiempo.

🔍 Anatomía de un Wireframe (Mid-fi)

✅ Debe incluir:

  • Estructura/Layout: Dónde va cada sección
  • Jerarquía: Qué es más importante (tamaños relativos)
  • Navegación: Menús, botones, breadcrumbs
  • Contenido placeholder: "Lorem ipsum" o texto genérico
  • Elementos UI: Cajas, botones, inputs, cards
  • Anotaciones: Notas sobre comportamiento

❌ NO debe incluir:

  • Colores (solo grises: #FFFFFF, #F5F5F5, #CCCCCC, #333333)
  • Fotos reales
  • Logos elaborados
  • Íconos detallados (usar placeholders)
  • Texto final/copy
  • Sombras complejas

✅❌ Ejemplos: Qué SÍ / NO en Wireframes

Elemento ❌ NO (Hi-fi) ✅ SÍ (Mid-fi)
Imagen de perfil Foto real del usuario Círculo gris con "👤"
Botón Botón azul #4A90E2 con sombra Rectángulo gris oscuro con "Botón"
Texto de título "Bienvenido a TaskSync, Laura" "Título de bienvenida"
Ícono Ícono detallado de calendar.svg Cuadrado con "📅" o "Icon"
Lista de items Texto real de 10 tareas "Item 1", "Item 2", "Item 3"

Regla: Si te tardas más de 2 minutos en un elemento, estás en demasiado detalle.

🔗 Del Sitemap a Wireframes

1Sitemap te dice QUÉ pantallas diseñar

Ejemplo: Sitemap tiene "Vista → Ver semanal" → Necesitas wireframe de pantalla "Ver semanal"

2Task Analysis (A3) te dice QUÉ va en cada pantalla

Ejemplo: Tarea "Revisar deadlines" → Pantalla debe mostrar lista de deadlines, filtros, fechas

3User Flows te dicen CÓMO conectar pantallas

Ejemplo: Usuario hace clic en tarea → Va a pantalla de detalle → Puede marcar completada

Fórmula: Sitemap (estructura) + A3 (contenido) + User Flows (navegación) = Wireframes completos

🚫 NO Hacer Hi-fi en Este Curso

Ejemplo Hi-fi (NO para DEIU):

  • Colores de marca (#4A90E2 para botones, #FF5555 para alertas)
  • Tipografía específica (Roboto 16px, Inter Bold 24px)
  • Fotos reales de usuarios
  • Íconos personalizados diseñados
  • Sombras, gradientes, efectos visuales
  • Texto final (copy específico)

¿Por qué NO hacer Hi-fi ahora?

  • Toma 3-4x más tiempo que Mid-fi
  • Dificulta iterar (ya invertiste mucho tiempo)
  • Usuarios se distraen con colores/fotos en vez de evaluar estructura
  • Primero validen estructura, LUEGO añadan estética

Para proyecto final: Pueden hacer Hi-fi AL FINAL, después de validar Mid-fi
